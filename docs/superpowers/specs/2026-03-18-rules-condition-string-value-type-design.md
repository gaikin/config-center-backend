# 智能提示条件配置值类型优化设计（统一字符串 + 数值运算智能比较）

## 1. 背景

当前智能提示「条件配置」中存在值类型（`STRING/NUMBER/BOOLEAN/...`）的显式配置，业务侧高频反馈问题是：

- 条件配置操作步骤偏多（需要额外关注值类型）
- 用户通常只关心“比较逻辑是否生效”，不想手动维护类型

本次目标是降低配置成本，保持规则表达能力，并避免字符串比较导致的典型误判。

## 2. 目标与边界

### 2.1 目标

- 条件值输入统一按字符串心智处理
- 去除值类型手工配置入口
- 对数值比较操作符（`GT/GE/LT/LE`）在运行时自动做数值转换并比较
- 不引入新的保存阻断规则（按“可保存、运行时不命中”策略）

### 2.2 非目标

- 不扩展日期/金额/枚举等新类型体系
- 不做历史数据迁移（新系统场景）
- 不改动 API 注册侧契约定义流程

## 3. 方案选型

采用方案 A：

- 配置层统一字符串
- `GT/GE/LT/LE` 执行时尝试数值转换
- 转换失败时条件判定为不命中，并输出可解释原因

不采用：

- 纯字符串大小比较（误判风险高）
- 保留值类型编辑（操作成本不降）

## 4. 行为设计

### 4.1 操作符行为矩阵

- `EQ/NE`：字符串比较
- `CONTAINS/NOT_CONTAINS/IN`：字符串语义
- `EXISTS`：保持现状
- `GT/GE/LT/LE`：
  - 将左右值执行 `Number(...)`
  - 双侧均为有限数值则按数值比较
  - 任一侧转换失败则该条件 `passed=false`，并记录原因

### 4.2 校验策略

- 保存时不增加“必须为数字”的阻断校验
- 仍保留“值是否为空”等基础校验
- 运行期给出失败原因，提升可解释性

## 5. 前端交互设计

### 5.1 条件配置面板

- 移除「值类型」可编辑区
- 不再要求用户手动选择 `RuleOperandValueType`
- 来源和值编辑保持不变

### 5.2 预览与执行反馈

在预览/执行 trace 中补充失败明细：

- “操作符为数值比较，但值无法转换为数字，按不命中处理”

## 6. 数据与兼容

- `RuleOperand.valueType` 字段暂保留（兼容结构）
- 比较逻辑不再依赖 `valueType` 决定数值运算
- 新创建/编辑规则默认走统一字符串心智

## 7. 实现清单

1. 页面交互调整  
   - 文件：`src/pages/RulesPage/RulesPage.tsx`  
   - 移除值类型编辑控件（或只读展示不再允许修改）

2. 页面保存校验调整  
   - 文件：`src/pages/RulesPage/useRulesPageModel.tsx`  
   - 不新增数值阻断，仅保留基础必填校验

3. 运行时比较逻辑调整  
   - 文件：`src/services/workflowService.ts`  
   - 对 `GT/GE/LT/LE` 引入数值转换分支与失败原因

4. 回归验证  
   - `EQ/NE/CONTAINS/IN/EXISTS` 行为不回退
   - `GT/GE/LT/LE` 数值有效与无效场景均可解释

## 8. 风险与缓解

### 风险

- 过去依赖 `valueType=NUMBER` 的隐式假设失效
- 若预览与执行链路实现不一致，可能出现口径偏差

### 缓解

- 统一收敛到 `workflowService` 的操作符语义
- 在预览链路与执行链路复用同一比较逻辑
- 增加针对操作符的回归用例

## 9. 验收标准

- 用户在条件配置中不再需要手工设置值类型
- `GT/GE/LT/LE` 面对非数字输入不报保存错误，但运行结果为不命中且有明确原因
- 其他操作符行为与现状一致
- 构建通过，关键流程手测通过
