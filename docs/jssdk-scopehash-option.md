# JSSDK scopeHash 方案说明（备选）

> 版本：v1.0  
> 日期：2026-03-18  
> 文档定位：备选方案说明  
> 使用时机：当同一菜单在不同机构、角色或灰度范围下返回配置内容不一致时

## 1. 目的

本文单独说明 `scopeHash` 的使用场景和设计边界，不作为当前 JSSDK 主方案文档的一部分。

## 2. 适用前提

只有在以下情况成立时，才需要考虑引入 `scopeHash`：

1. 同一菜单在不同机构下返回配置不同。
2. 同一菜单在不同角色下返回配置不同。
3. 同一菜单在不同灰度命中范围下返回配置不同。

如果同一菜单对所有用户返回一致配置，则不需要 `scopeHash`。

## 3. 方案定位

`scopeHash` 不是版本号。

它的职责只有一个：

`作为缓存隔离键，防止不同范围下错误复用同一份本地缓存`

## 4. 与 menuVersion 的关系

1. `menuVersion` 仍然只按菜单维护。
2. `scopeHash` 不参与版本治理。
3. 菜单有变更时，仍然统一升级 `menuVersion`。
4. `scopeHash` 只影响“本地缓存怎么隔离”，不影响“版本号怎么升级”。

## 5. 推荐缓存键

```text
cc:jssdk:menu:{menuCode}:{menuVersion}:{scopeHash}:{schemaVersion}
```

## 6. 风险

如果应该隔离却没有引入 `scopeHash`，可能发生：

1. A 机构缓存的配置被 B 机构错误复用。
2. 角色差异配置发生串缓存。
3. 灰度试点配置被非试点用户错误命中。

## 7. 当前建议

1. 主方案文档先不展开 `scopeHash`。
2. 若后续确认配置结果存在范围差异，再单独评审并接入该方案。
